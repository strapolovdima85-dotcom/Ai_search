# Архитектура системы: Интеллектуальный поиск номенклатуры в 1С:ERP

## 1. Общая концепция и паттерн интеграции
Для минимизации нагрузки на транзакционную базу данных 1С:ERP применяется **асинхронная гибридная архитектура**. Тяжелые вычисления (генерация эмбеддингов, векторный поиск) выносятся за пределы 1С в специализированный ИИ-микросервис на Python.

## 2. Компоненты архитектуры

*   **Клиентский слой (1С:ERP):** Внешняя обработка или расширение формы подбора. Отвечает только за ввод строки поиска, отображение результатов и финальное оформление заказа.
*   **Слой бизнес-логики (AI-Search Service на Python):** FastAPI-приложение. Принимает текстовый запрос от 1С, проводит его предобработку, векторизует и управляет логикой гибридного поиска.
*   **Слой ИИ-моделей (Embedding & Ranking):** 
    *   *Bi-Encoder (например, RuBERT-tiny / LaBSE):* Для быстрого перевода текста в вектор (эмбеддинг).
    *   *Cross-Encoder (опционально):* Для финального точного реранжирования ТОП-20 результатов.
*   **Слой хранения векторов (Vector DB):** Специализированная база данных (например, Qdrant или расширение pgvector для PostgreSQL) для быстрого поиска ближайших соседей (HNSW индекс).

## 3. Сценарии взаимодействия (Data Flow)

### 3.1. Сценарий А: Первоначальное наполнение и синхронизация (Offline/Батч)
1. 1С:ERP выгружает массив номенклатуры (ID, Наименование, Описание, Артикул, Характеристики) в Python-сервис.
2. Сервис генерирует семантический вектор (эмбеддинг) для каждой позиции.
3. Векторы вместе с метаданными (ID 1C) сохраняются в Векторную БД.
4. *При изменении/добавлении позиции в 1С:* Срабатывает подписка на событие, отправляя в ИИ-сервис только измененную карточку (инкрементальное обновление).

### 3.2. Сценарий Б: Выполнение поискового запроса менеджером (Online/Real-time)
1. Менеджер вводит в строке подбора 1С: `«черный саморез 3.5х19»`.
2. 1С отправляет HTTP POST запрос на эндпоинт `/search`.
3. Python-сервис очищает строку и пропускает её через Embedding-модель, получая вектор запроса.
4. Выполняется поиск в Векторной БД по алгоритму приближенных ближайших соседей (ANN).
5. ИИ-сервис возвращает массив вида `[{id_1c: "uuid-1", score: 0.94}, {id_1c: "uuid-2", score: 0.89}]`.
6. 1С получает JSON, мгновенно позиционирует фокус на найденных товарах в динамическом списке и подтягивает актуальные остатки/цены.

## 4. Стек технологий

*   **Backend:** Python 3.11+, FastAPI (асинхронная обработка запросов).
*   **Vector Database:** Qdrant (высокая скорость, встроенная поддержка гибридного поиска).
*   **Machine Learning / NLP:** Hugging Face Transformers, PyTorch.
*   **Базовая модель:** `sentence-transformers/rubert-base-cased-sentence` или кастомная легковесная модель, дообученная на текстах номенклатуры предприятия.
*   **Протокол обмена:** REST API (JSON).

## 5. Метрики эффективности архитектуры (SLA / KPI)
*   **Время отклика (Latency):** < 250 мс на запрос.
*   **Надежность (Availability):** Fallback на стандартный полнотекстовый поиск 1С при недоступности ИИ-сервиса (сбоев в бизнес-процессе продаж быть не должно).
*   **Релевантность (Качество ИИ):** MRR (Mean Reciprocal Rank) > 0.85 на тестовой выборке запросов от реальных пользователей.
